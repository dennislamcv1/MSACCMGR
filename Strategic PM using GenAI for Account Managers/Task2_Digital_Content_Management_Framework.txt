━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This Digital Content Management Framework (DCMF) governs how project
information is created, stored, classified, retrieved, shared, and retired
across the 15-month HealthSystem Regional engagement. It is designed to serve
three simultaneous organizational objectives:

  OPERATIONAL EXCELLENCE
  Ensure that every project team member — regardless of location or role —
  can find the right document, in the right version, at the right time.
  Reduce time lost to document search, version confusion, and information
  silos that degrade delivery performance.

  STRATEGIC INTELLIGENCE
  Structure project knowledge so that insights, decisions, and outcomes
  accumulated during this $3.2M engagement are preserved as institutional
  intelligence that informs CloudTech's pursuit of the $15M program expansion
  and future healthcare consulting engagements.

  COMPLIANCE & TRUST
  Demonstrate to HealthSystem Regional that CloudTech manages sensitive
  project information with the rigor expected of a long-term digital
  transformation partner, meeting HIPAA, HITECH, and enterprise information
  security standards at every level.

The framework is built on four integrated pillars:
  Pillar A — SharePoint Organization Model (Structure)
  Pillar B — Metadata Schema Design (Classification)
  Pillar C — Content Category Mapping (Taxonomy)
  Pillar D — Governance, Compliance & Search Optimization (Control & Discovery)

ARCHITECTURE OVERVIEW
----------------------
The SharePoint environment is structured as a two-tier site architecture:
a top-level Program Hub and an Engagement-Specific Project Site. This design
enables cross-program knowledge sharing at the portfolio level while maintaining
security and access controls at the individual engagement level.

  TIER 1 | PROGRAM HUB SITE
  Name:    CloudTech Healthcare Transformation Program Hub
  URL:     /sites/CT-HealthTransform-Hub
  Purpose: Portfolio-level visibility; strategic intelligence repository;
           cross-engagement learning; executive relationship management assets.
  Access:  CloudTech leadership, Engagement Directors, Strategic PM leads.
           HealthSystem Regional does NOT have access to this tier.

  TIER 2 | ENGAGEMENT PROJECT SITE
  Name:    HSR Patient Engagement Platform — Project Site
  URL:     /sites/CT-HSR-PEP-2025
  Purpose: Day-to-day project execution; client-facing document collaboration;
           governance records; compliance documentation.
  Access:  CloudTech project team + designated HealthSystem Regional
           stakeholders (role-based, as defined in Section 4).

METADATA PHILOSOPHY
--------------------
Metadata is the intelligence layer that transforms a folder structure from a
static filing cabinet into a dynamic, searchable, and reportable knowledge
system. Where folder structures organize content by location, metadata
classifies content by meaning — enabling documents to be found not just by
who knows where they are filed, but by what they are, what they relate to,
and when they matter.

This schema is designed to support four metadata-driven capabilities:
  (1) Precision Search — find the right document in seconds without knowing
      its exact name or location.
  (2) Cross-Library Reporting — surface all documents related to a phase,
      risk, or stakeholder regardless of which library they live in.
  (3) Compliance Auditing — generate audit-ready reports of all documents
      by classification, access tier, or regulatory category.
  (4) Strategic Intelligence Mining — identify patterns across document
      types and engagement phases to inform future program strategy.

────────────────────────────────────────────────────────────────────────────────
CORE METADATA SCHEMA — UNIVERSAL FIELDS
(Applied to ALL documents across ALL libraries)
────────────────────────────────────────────────────────────────────────────────

FIELD 1: Document Type
  Data Type:    Managed Metadata (Term Store)
  Values:       Plan | Report | Charter | Specification | Minutes | Assessment |
                Briefing | Roadmap | Change Record | Training Material |
                Correspondence | Decision Record | Risk Register | Invoice |
                Dashboard | Lessons Learned
  Rationale:    Enables filtering across libraries by document category.
                Supports governance reporting ("show me all risk registers
                created in Phase 3") and audit trail generation.

FIELD 2: Project Phase
  Data Type:    Choice (Single Select)
  Values:       Phase 0 – Pre-Engagement | Phase 1 – Discovery & Alignment |
                Phase 2 – Design & Configuration | Phase 3 – Pilot Deployment |
                Phase 4 – Enterprise Rollout | Phase 5 – Optimization & Handoff |
                Program Level (Cross-Phase)
  Rationale:    Allows time-based filtering and phase-specific reporting.
                Essential for Gate Review documentation packages and retrospective
                analysis. Supports Phase 2 business case preparation by enabling
                retrieval of all Phase 1–5 strategic outputs in one query.

FIELD 3: Stakeholder Audience
  Data Type:    Managed Metadata (Multi-Select)
  Values:       Internal – CloudTech Only | Executive (C-Suite) |
                Clinical Leadership | Operational / IT | All Stakeholders |
                Regulatory / Legal | External Partner
  Rationale:    Enables rapid assembly of stakeholder-specific document packages
                (e.g., "compile all documents tagged for Executive audience
                in Phase 3"). Supports communication planning and access control
                decisions.

FIELD 4: Confidentiality Classification
  Data Type:    Choice (Single Select, Required)
  Values:       PUBLIC – Shareable externally |
                INTERNAL – CloudTech team only |
                CONFIDENTIAL – CloudTech + named HS Regional stakeholders |
                RESTRICTED – Senior leadership only |
                REGULATED – Contains HIPAA-adjacent or legally sensitive content
  Rationale:    Drives automated permission recommendations and data loss
                prevention (DLP) policy application. Ensures compliance with
                HIPAA and HS Regional's Information Security Policy. Required
                field to prevent ungoverned document sharing.

FIELD 5: Document Status
  Data Type:    Choice (Single Select)
  Values:       Draft | In Review | Approved | Superseded | Archived | Void
  Rationale:    Prevents stakeholders from acting on draft or superseded
                documents. Combined with version control, creates a full
                document lifecycle record. Supports quality management and
                audit requirements.

FIELD 6: Document Owner
  Data Type:    Person / Group (Single)
  Values:       [CloudTech team member responsible for content accuracy]
  Rationale:    Establishes accountability for document quality and review
                cycles. Enables automated review reminders via Power Automate.
                Provides a clear point of contact for document questions.

FIELD 7: Review / Expiry Date
  Data Type:    Date (Required for Plans, Policies, Risk Registers)
  Rationale:    Triggers automated review notifications before documents
                become stale. Particularly important for Risk Registers,
                Compliance Records, and Project Plans that require regular
                updates to remain accurate.

FIELD 8: Related Risk ID
  Data Type:    Text (Multi-Value, Optional)
  Values:       [Risk Register ID, e.g., RISK-001, RISK-014]
  Rationale:    Creates a cross-document risk traceability chain. Allows
                the PM to query "show me every document related to RISK-003
                (EHR Integration Complexity)" and instantly surface all
                mitigation plans, meeting notes, change records, and status
                reports related to that risk.

GOVERNANCE PRINCIPLES
----------------------
Content governance on this engagement operates under the principle of least
privilege — every user has access to exactly what they need, and no more.
This principle protects HS Regional's sensitive organizational data, CloudTech's
proprietary strategic intelligence, and the integrity of the contractual record.

────────────────────────────────────────────────────────────────────────────────
4A. VERSION CONTROL STANDARDS
────────────────────────────────────────────────────────────────────────────────

AUTOMATED VERSIONING
  - SharePoint major and minor version history is enabled across all libraries.
  - Major versions (1.0, 2.0, 3.0) represent formally approved document states.
  - Minor versions (1.1, 1.2, 1.3) represent working drafts under review.
  - Version history is retained indefinitely for all documents in Category C
    (Client Communications & Change Records); 3 years for all other categories.

VERSION NAMING CONVENTION
  Standard format: [DocumentName]_v[Major].[Minor]_[YYYYMMDD]
  Examples:
    ProjectManagementPlan_v1.0_20250201   — Initial approved baseline
    ProjectManagementPlan_v1.2_20250315   — Draft revision under review
    ProjectManagementPlan_v2.0_20250401   — Updated approved version
    EHRIntegrationSpec_v1.0_20250310      — First approved integration spec

CO-AUTHORING CONTROLS
  - Microsoft 365 co-authoring is enabled for draft documents (minor versions).
  - Document check-out is required before editing approved documents (major
    versions), preventing simultaneous conflicting edits.
  - All edits to RESTRICTED documents require check-out regardless of status.

CHANGE FREEZE PROTOCOL
  - At each Gate Review, a "Configuration Freeze" is applied to all documents
    supporting that gate. Frozen documents cannot be edited without a formal
    Document Change Request, logged in the Change Management Library.

ROLE-BASED ACCESS CONTROL (RBAC) MATRIX
  Access is managed through SharePoint Permission Groups, not individual user
  assignments, to ensure consistent control as team membership changes.

  ROLE GROUP 1: CloudTech Core Project Team
    Access Level: Full Control on Libraries 1–5
    Members:      Project Manager, Technical Lead, Change Manager,
                  Integration Architect, QA Lead
    Justification: Operational delivery team requires full read/write access
                   across all project content.

  ROLE GROUP 2: CloudTech Leadership / Program Oversight
    Access Level: Read + Contribute on Libraries 1–5;
                  Full Control on Library 3 (Strategic Intelligence)
    Members:      Engagement Director, Portfolio Manager, Practice Lead
    Justification: Leadership oversight access without day-to-day edit
                   permissions; full control of strategic intelligence content
                   that is CloudTech-proprietary.

  ROLE GROUP 3: HS Regional Executive Stakeholders
    Access Level: Read Only — Libraries 4 (selected folders) and 5
    Members:      HS Regional CIO, CMO, Designated Executive Assistants
    Allowed:      Executive Briefings, Status Reports, Performance Dashboards,
                  Decision Log (read), Approved Training Materials
    Denied:       Library 1 (financial details), Library 3 (ALL — strategic
                  intelligence is internal CloudTech only), Change Request
                  details prior to formal approval

  ROLE GROUP 4: HS Regional Clinical & Technical Stakeholders
    Access Level: Read + Contribute (designated folders only)
    Members:      IT Directors, Clinical Advisory Board members,
                  Training Coordinators, Facility Operations Managers
    Allowed:      Technical Specifications (review comments), UAT Records,
                  Training Materials (contribute for facility-specific content),
                  Meeting Minutes (read)
    Denied:       Libraries 1, 3; Budget documents; Escalation Records

  ROLE GROUP 5: Read-Only Observers
    Access Level: Read Only — Library 4.2 and Library 5.1 only
    Members:      Facility Department Heads, HR Training Staff
    Justification: Visibility into training materials and status updates
                   without broader project access.

EXTERNAL SHARING CONTROLS
  - External sharing is disabled at the tenant level for Library 3
    (Strategic Intelligence) and all RESTRICTED documents.
  - Sharing links for CONFIDENTIAL documents require internal authentication
    (no anonymous links).
  - All external sharing activity is logged and reviewed weekly by the PM.

PRIVILEGED ACCESS REVIEW
  - Access group membership is reviewed monthly by the PM.
  - When team members depart (CloudTech or HS Regional), access is revoked
    within 4 business hours via the Offboarding Access Revocation Checklist.

4C. HEALTHCARE DATA PROTECTION & HIPAA COMPLIANCE CONTROLS
────────────────────────────────────────────────────────────────────────────────

ZERO PHI POLICY ENFORCEMENT
  - No Protected Health Information (PHI) is to be stored in the SharePoint
    environment under any circumstances. The project site manages project
    documentation, not clinical data.
  - If a document inadvertently contains PHI (e.g., a training scenario using
    real patient data), the Document Owner must immediately:
    (a) Remove access to the document
    (b) Notify the PM within 1 hour
    (c) Initiate the PHI Incident Response Protocol
    (d) Replace the document with a de-identified version

DATA LOSS PREVENTION (DLP) POLICIES
  - Microsoft Purview DLP policies are configured to automatically scan all
    uploads for patterns that may indicate PHI (SSNs, DOBs, MRNs, etc.).
  - Documents triggering DLP alerts are automatically quarantined pending
    PM review and reclassification or remediation.

ENCRYPTION STANDARDS
  - All documents at rest: AES-256 encryption (Microsoft 365 default).
  - All documents in transit: TLS 1.3 minimum.
  - RESTRICTED documents: Microsoft Purview Information Protection labels
    applied, enabling persistent encryption and access tracking even when
    documents are downloaded.

AUDIT LOGGING
  - SharePoint Unified Audit Logs capture all document access, edit, download,
    share, and delete events.
  - Audit logs are retained for 7 years, aligned with HIPAA audit requirements
    and HS Regional's IS policy.
  - Monthly audit log reviews are conducted by the PM; anomalous access events
    (downloads of RESTRICTED documents, access outside business hours)
    trigger automated alerts to the Engagement Director.

RETENTION & DISPOSAL POLICY
  Document Category         | Active Retention | Post-Project Retention
  ————————————————————————————————————————————————
  Project Planning / Tech   | Project duration  | 3 years
  Strategic Intelligence    | Project + 5 yrs   | 5 years (CloudTech IP)
  Client Comms / Contracts  | Project duration  | 7 years (legal requirement)
  Training / Performance    | Project + 2 yrs   | 2 years
  ————————————————————————————————————————————————
  Disposal Method: Documents are not deleted — they are moved to a
  read-only archive site with retained audit history. Physical destruction
  of any printed copies follows NIST SP 800-88 media sanitization standards.

THE LIMITATIONS OF FOLDER-ONLY ARCHITECTURE
---------------------------------------------
Traditional document management systems organize content exclusively through
hierarchical folder structures. While intuitive to set up, folder-only systems
create significant findability and governance problems as projects scale:

  PROBLEM 1 — THE "ONE LOCATION" CONSTRAINT
  A document can only exist in one folder. A Risk Register that is equally
  relevant to Phase 2, the Executive stakeholder audience, and the EHR
  Integration risk stream must be placed in a single location — making it
  invisible to the other two use contexts unless copies are created.
  Copies breed version confusion: which one is current?

  PROBLEM 2 — RETRIEVAL REQUIRES LOCATION KNOWLEDGE
  Finding a document in a folder structure requires knowing where it was
  filed. New team members, executive stakeholders, or cross-functional
  collaborators who don't know the filing convention spend significant
  time searching manually — or simply ask the PM, consuming skilled time
  on administrative tasks.

  PROBLEM 3 — REPORTING IS MANUAL
  Generating a compliance report of all HIPAA-sensitive documents, or a
  Phase 3 document package for a Gate Review, requires manually opening
  folders and selecting files. There is no automated way to query content
  by multiple dimensions simultaneously.

  PROBLEM 4 — GOVERNANCE GAPS EMERGE
  Without mandatory classification fields, documents are frequently
  misfiled, left without version numbers, or shared with incorrect
  audiences. Governance relies entirely on human discipline — which
  degrades under project delivery pressure.

METADATA-DRIVEN SEARCH: HOW IT WORKS
--------------------------------------
In this framework, SharePoint's managed metadata and search capabilities
transform document retrieval from a location-based hunt into a multi-
dimensional query. The mechanics work as follows:

  STEP 1 — UPLOAD WITH MANDATORY METADATA
  When a document is uploaded, SharePoint requires the user to complete
  the defined metadata fields before the document is published. This is
  enforced through column validation rules, preventing ungoverned uploads.

  STEP 2 — MICROSOFT SEARCH INDEXING
  Microsoft Search indexes both document content (full text) and metadata
  fields simultaneously. A single document becomes discoverable by its
  name, its content, its phase, its type, its audience, its status,
  and its related risk — all independently searchable dimensions.

  STEP 3 — QUERY WITHOUT LOCATION KNOWLEDGE
  A team member searching for "all approved documents for Phase 3 Clinical
  Advisory Board" can filter by:
    Document Status = Approved
    Project Phase = Phase 3 – Pilot Deployment
    Meeting Type = Clinical Advisory Board
  ...and receive all matching documents across all libraries in seconds,
  without knowing the folder structure.

MICROSOFT COPILOT INTEGRATION WITH METADATA SEARCH
----------------------------------------------------
This metadata architecture is specifically designed to maximize the value of
Microsoft Copilot's enterprise search and content summarization capabilities:

  COPILOT FOR MICROSOFT 365 — PROJECT SEARCH
  Copilot can process natural-language project queries ("What decisions were
  made in the Phase 2 Gate Review related to EHR integration?") and retrieve
  relevant documents from across all libraries, using metadata as the
  precision filter that ensures accurate results rather than broad text
  matches. Without rich metadata, Copilot returns low-precision results.
  With this schema, Copilot becomes a true project intelligence assistant.

  COPILOT REPORT GENERATION
  Monthly Executive Briefings can be generated by Copilot by querying
  "all KPI dashboard documents tagged Metric Category = Patient Engagement,
  Project Phase = Phase 3, Document Status = Approved" — producing
  source-verified drafts in minutes rather than hours.

SEARCH OPTIMIZATION CONFIGURATION IN SHAREPOINT
-------------------------------------------------
  - Custom search verticals are configured for each Content Category,
    allowing users to search within a specific domain (e.g., "Technical
    Documentation" or "Executive Briefings") rather than the entire site.
  - Refinement panels expose all metadata fields as search filters,
    enabling rapid narrowing of results.
  - Promoted results are configured for the most frequently accessed
    documents (Master Schedule, Risk Register, KPI Dashboard) so they
    surface as top results for common queries.
  - Saved search views are published for recurring governance use cases:
      "Phase [N] Gate Review Document Package"
      "This Week's Executive-Audience Documents"
      "All Open Action Items — Decision Log"
      "Training Materials by Role — [Role Name]"
